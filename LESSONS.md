## 技術的教訓

### L1: file.Appendは.txt拡張子のみ対応
- **発見セッション**: Pre-Template Era
- **内容**: `file.Append(".log")`等は無視される。エラーも出ない（サイレント失敗）
- **対策**: ログファイルも必ず`.txt`拡張子で作成する

### L2: string.StartWithはsなし
- **発見セッション**: Pre-Template Era
- **内容**: GLuaでは`string.StartWith`が正しいAPI。`string.StartsWith`（sあり）はエラーになる
- **対策**: 常に`string.StartWith`を使用。他のLuaフレームワークとの混同に注意

### L3: ConVarは毎フレーム取得禁止
- **発見セッション**: Pre-Template Era
- **内容**: `GetConVar()`を毎フレーム呼ぶとパフォーマンスに影響する
- **対策**: ローカル変数で遅延初期化パターンを使用: `local cv = cv or GetConVar("vrmod_unoff_*")`

### L4: C++モジュール関数はpcall必須
- **発見セッション**: Pre-Template Era
- **内容**: VRModのC++モジュール関数（`vrmod.GetHMDPos()`等）はVR未接続時にエラーを投げる可能性がある
- **対策**: 必ず`pcall()`で保護する。戻り値のチェックも行う

### L5: 防御的初期化パターン
- **発見セッション**: Pre-Template Era
- **内容**: `vrmod`テーブルの子テーブルが他のファイルで未定義の場合、nilアクセスでエラーになる
- **対策**: `vrmod.X = vrmod.X or {}` で防御的に初期化する

## アーキテクチャ教訓

### L6: LVS車両統合は不安定になりやすい
- **発見セッション**: Pre-Template Era
- **内容**: LVS車両のServer処理変更が複数回ロールバックされた（Gitログ: 3fd43df, 3b0c40b）
- **対策**: LVS関連の変更は小さなステップで行い、各ステップでマルチプレイテストを実施する

### L7: カスタムHUDは禁止方針
- **発見セッション**: Pre-Template Era
- **内容**: 独自HUD要素を描画するとVRModの既存HUDと競合する。4ファイルを削除した経緯がある
- **対策**: 既存HUDをRT→VR手に表示する方式のみ使用。新規HUD描画は行わない

### L8: Melee速度計算は相対速度を使うべき
- **発見セッション**: x64比較調査 (2026-02-27)
- **内容**: `vrmod.GetRightHandVelocity()` は絶対世界速度を返す。走行速度400 u/sで手を静止していても `400/40 = 10.0` となり、最大ダメージtier（閾値4.35）を常に超過する
- **対策**: `hand_velocity - vrmod.GetHMDVelocity()` で相対速度を算出。HMD速度を差し引くことで純粋な振りの速さのみを検出

### L9: x64の技術的優位は実用性とは別
- **発見セッション**: x64比較調査 (2026-02-27)
- **内容**: x64の連続ダメージスケール、武器自動認識、コリジョン押し戻しは技術的に洗練されているが、Gmod VRの実際のゲームプレイでは不適切。連続スケールは格ゲー的な弱中強の使い分けを破壊し、武器認識はHL2限定、コリジョン押し戻しはGmodの不安定な物理で逆効果
- **対策**: x64から採用するのは相対速度計算とModderフックのみ。「技術的に優れている＝実用的に優れている」ではない

### L10: VRMod_Trackingはフレームベースで発火する
- **発見セッション**: x64比較調査 (2026-02-27)
- **内容**: `VRMod_Tracking` hookはRenderScene内で発火するため、フレームレートに依存する。120FPSなら120回/秒、30FPSなら30回/秒。低FPSで高速移動時にターゲットをすり抜ける可能性がある
- **対策**: 現時点では根本的解決は困難（連続スイープトレースが必要）。将来的な改善候補として記録

## プロセス教訓

### L11: x64との比較は選択的に行う
- **発見セッション**: x64比較調査 (2026-02-27)
- **内容**: x64の「改善」を無批判に取り込むと設計意図を壊す。各要素を実際のゲームプレイ観点で評価し、採用/不採用を判断すべき
- **対策**: 技術的優位だけでなく、Gmod VRの現実的な使用シナリオで評価する

### L12: グローバル関数の二重ラップは循環参照を起こす
- **発見セッション**: S2.5 HUD Stack Overflow緊急修正 (2026-02-27)
- **内容**: 複数ファイルが同じグローバル関数（`VRUtilRenderMenuSystem`）を独立に`orig = orig or GlobalFunc`パターンでラップすると、ロード順やreincludeタイミングにより`orig`が他のラッパーを捕捉し循環参照→スタックオーバーフローになる
- **対策**: 真のオリジナル関数は`vrmod._origXxx`のようにテーブルに保存し（canonical original pattern）、各ラッパーはそこから参照する。防御策として再入ガード（`_in_render`フラグ）も併用

### L13: 大規模UI改善は別タスクとして分離する
- **発見セッション**: S8 Mirror UI調査（2026-03-01）
- **内容**: タスク5「Mirror UIの整理」調査中、ユーザーの7年間の経験から「ウィザード形式セットアップ」という理想的UI設計が提案された。しかし、これは元のタスク（重複ボタン削除）を大きく超える大規模実装となるため、実装は保留
- **対策**: 1セッション1〜2タスクの原則を守る。大規模アイデアは将来の独立タスクとしてBACKLOGに記録し、小さく確実に進める方針を維持する

### L14: ユーザーの実際の使用手順を文書化する価値
- **発見セッション**: S8 Mirror UI調査（2026-03-01）
- **内容**: Mirror UIの問題点を調査する際、ユーザーの実際の手順（キャラクター変更時：Auto Set 3回→Auto Scale 3回→微調整→Dist調整→Hide Head）を聞くことで、Auto ScaleとAuto Setの両方が必要で使用順序も重要であることが判明。単純な統合ではなく、ウィザード形式という理想的解決策に繋がった
- **対策**: UI改善タスクでは、実装前にユーザーの実際の操作手順を詳細に聞き取り、文書化する。理想論ではなく実際の使用パターンから設計する

### L15: PDCAサイクルはセッション終了チェックリストより有効
- **発見セッション**: S9 PDCAワークフロー設計（2026-03-01）
- **内容**: CODE_STATUS.mdの更新がS6-S8で抜けていた。「毎セッション終了時の重厚なチェックリスト」は過去の失敗パターン（vrmod_semioffcial_re）を繰り返す。代わりにPDCAサイクル（4専用会話）+ 定期大規模点検（5-10タスクごと）で品質管理する。トークン枯渇防止が最重要制約
- **対策**: 1タスク=4専用会話（P→D→C→A）で分離。Check会話でユーザーが採点し、60点未満ならAction会話で修正。大規模なメタ更新は定期点検で一括実施
